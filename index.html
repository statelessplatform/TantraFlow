Why TantraFlow

The problems we kept running into

Every AI tool promised to "automate your workflow" — but none of them actually handled the messy, multi-step, context-passing reality of real knowledge work.

Cloud lock-in and data leakage
Sending client research, financial data, or internal documents to OpenAI or Anthropic APIs is a compliance risk for most businesses. Every token you send is data you no longer fully control. TantraFlow runs entirely on your hardware — nothing leaves your machine.
Single-agent limitations
One LLM call can write, but it can't also research, then structure, then format, then send. Context windows fill up. Quality degrades. TantraFlow decomposes tasks across specialist agents — each expert in their domain, each receiving focused context.
No output accountability
When an AI pipeline fails, you rarely know which step went wrong, what context it received, or how long it spent. TantraFlow logs every node execution, writes per-node output files, and gives you a full trace with latency and token counts.
Stitching tools together
Teams were copying ChatGPT output into Google Docs, then into PDFs, then emailing manually. 20 minutes of "AI work" plus 40 minutes of manual formatting. TantraFlow automates the entire chain — from prompt to polished deliverable — in one workflow execution.
No repeatability
A great prompt in ChatGPT disappears the moment the browser closes. TantraFlow stores your agent definitions, workflow graphs, and every execution's outputs permanently. Run the same market research workflow next month with one click.
Enterprise needs, startup budgets
Enterprise AI platforms charge thousands per month for workflow automation. TantraFlow is open source, self-hosted, and runs on a MacBook or a $20/month VPS. The only cost is the electricity to run your models.
Key Learnings

What building this taught us

1

Context quality beats context quantity

Passing an entire previous agent's 4,000-word output to the next agent causes ReadTimeout and poor quality. Truncating to 6,000 characters of the most relevant content dramatically improved results.

2

Cold-start is the silent killer

On Apple Silicon, Ollama takes 30–90 seconds to load a model from disk. Without a warmup ping before the first node, every workflow's first node timed out at 120 seconds. A tiny pre-flight request fixes this entirely.

3

LLMs always ignore output format instructions

Even when told "output ONLY raw HTML", models wrap output in markdown fences. Every output handler now strips fences before writing files. This turned broken HTML files into clean, renderable documents.

4

Specialist agents outperform generalists

A "Research Agent" with a focused skills.md produces better research than a generic agent told to "do research". Role definition, behavioral guidelines, and output format instructions are the real prompt engineering.

5

Smaller models are better for most tasks

llama3.2:3b completes a 4-node workflow in under 3 minutes on an M2. llama3.1:70b takes 15+ minutes and hits timeouts. For most document generation tasks, 3B parameters is enough — speed wins.

6

The UI is the product

A technically brilliant backend nobody can use delivers zero value. The visual builder, live execution monitor, and one-click "Create Agent from Template" features are what make TantraFlow actually usable for non-developers.

A multi-agent pipeline orchestrator

Visual Workflow Builder

Drag specialist agents onto a canvas, connect them with edges to define data flow. Each node produces a typed output file (.txt, .html, .pdf, .py, .sh, .bat, .csv, .json) that feeds the next node as context.

Specialist Agent System

Each agent has a skills.md defining its role, capabilities, and behavioral guidelines, plus a tools.py with callable functions. Built-in: Orchestrator, Researcher, Writer, Web Designer, PDF Formatter, Code Reviewer, Data Analyst, Email Agent, SysAdmin, QA Tester.

Live Execution Monitor

Watch each node process in real time. See model, latency, token count, output file type, and any errors — per node. Stop a running workflow at any time; partial outputs remain available for download.

Import / Export Bundles

Share a complete workflow — including all agent definitions, skills, and tools — as a single JSON file. Your colleague imports it on their instance and runs it immediately, with their own local models.

How to Use TantraFlow

From zero to first workflow in 10 minutes

Follow these steps in order. By Step 6 you will have run your first real multi-agent pipeline and downloaded its output files.

1

Install and start TantraFlow

Clone the repo, run the start script. It creates a Python virtualenv, installs dependencies, and launches the FastAPI server on port 8000.

git clone https://github.com/statelessplatform/TantraFlow agentic-platform cd agentic-platform chmod +x start.sh ./start.sh # macOS / Linux # start.bat # Windows # You should see: # ✓ Database ready → SQLite # ✓ Seeded 10 library workflow(s) # Uvicorn running on http://0.0.0.0:8000
The start script requires Python 3.10–3.13. It will tell you if your version is wrong. Python 3.14+ is not supported.
2

Connect your local LLM

TantraFlow calls your own LLM — it doesn't have API keys. Install Ollama (recommended) or LM Studio, then add it as a Model Source.

# Install Ollama (macOS/Linux) curl -fsSL https://ollama.com/install.sh | sh ollama serve # start the server # Pull a model (choose based on your RAM) ollama pull llama3.2 # 3B — fast, 4GB RAM ollama pull mistral # 7B — balanced, 8GB RAM ollama pull llama3.1:70b # 70B — quality, 48GB RAM

Then in TantraFlow: Models → Add Source → Type: Ollama · URL: http://localhost:11434

The sidebar shows a green dot when TantraFlow can reach your LLM. Set your default model in Settings — it applies to all agents that don't have their own model assigned.
3

Browse the Workflow Library

Navigate to Workflows → Library tab. You'll see 10 production-ready workflows, each with a documentation button explaining its pipeline and output files.

Click Docs on any workflow card to read what it produces, how many agents are involved, and what the output files will be.

Start with Market Research Report (4 nodes → PDF) or Software Project Kickstart (5 nodes → .py + .sh + tests). These are the most complete and produce immediately useful files.
4

Assign a model to each agent

Before running a workflow, tell each agent which model to use. Navigate to Agents — you'll see 10 agents were seeded. Click the model dropdown on any agent card to assign it.

The fastest option: set a global default in Settings → Default LLM Model. All agents with no explicit model assigned will use this fallback.

For fastest results, assign llama3.2 (3B) to all agents. Once you're comfortable, assign larger models to specific agents — for example, use mistral:7b for the Writer Agent for better prose quality.
5

Run your first workflow

Back on the Workflows page, click Run on any workflow card. A modal opens with a task input field. Type what you want the workflow to produce — be specific.

# Example inputs: # Market Research Report: "Analyse the competitive landscape for AI-powered project management tools. Focus on pricing, target audience, and key differentiators." # Software Project Kickstart: "A FastAPI REST API for a task management system with PostgreSQL, JWT authentication, and CRUD for tasks and users." # Code Review: "Review this Python function for security and performance issues: def get_user(id): return db.execute(f'SELECT * FROM users WHERE id={id}')"

Click Execute Now and watch the live monitor — each node lights up green as it completes. You can click Stop at any time to halt execution after the current node.

6

Download your output files

When execution completes, the output files appear inline in the modal with download links. You can also find them on the Logs page by clicking any execution row to open the monitor.

All files are saved locally to data/outputs/<trace_id>/. Open HTML files directly in your browser. PDF files require WeasyPrint to be installed (pip install weasyprint).

7

Build your own workflow

Go to Builder. Select a workflow from the dropdown to load it into the canvas, or start fresh. Drag agents from the left palette onto the canvas, then click node ports to draw connection arrows.

Click any node to open its Properties panel on the right — change the label, assign a specific agent, set the output type, and override the model for that node specifically.

Connection arrows show data flow direction. Node 1's output becomes Node 2's context. Set output types thoughtfully — a Research node should output .txt, while a Web Designer node should output .html.
8

Create custom agents with Skills

Navigate to Skills & Tools. The Template Library tab has 8 built-in templates (Research, Writer, Web Designer, PDF Formatter, and more). Click Save to add one to your library, or Create Agent to instantiate it immediately.

To create a custom skill: click New Skill → fill in the skills.md (role, capabilities, guidelines) and tools.py (callable functions) → Save → Create Agent → assign a model on the Agents page.

9

Share workflows with your team

On any workflow card, click the Export button. TantraFlow downloads a .json bundle containing the complete workflow definition plus all agent skills and tools. Your teammate imports it on their instance with one click.

# Your colleague runs: # Workflows → Import → paste the JSON bundle # The import: # ✓ Creates agents (or updates existing ones by name) # ✓ Remaps agent IDs to their local instance # ✓ Inserts the workflow ready to run
10

Monitor, iterate, improve

The Logs page shows every execution event with timestamps and token counts. Click any execution card in the strip at the top to open the live monitor — even for completed executions.

When a node produces poor output, open its output file, read what it actually received as context (in the combined_output.md), and refine the agent's skills.md. Small changes to role definition and behavioral guidelines have the largest impact on output quality.

Pro tip: Add "NEVER truncate — produce the complete document" to every agent's behavioral guidelines. This single instruction eliminates 80% of incomplete output issues.
FAQ

Common questions

Does TantraFlow send my data to any cloud service?
No. TantraFlow runs entirely on your machine. It calls only the LLM server you configure (Ollama or LM Studio, both local). The only external requests are to Google Fonts for typography and Font Awesome for icons — no data leaves your network.
Which models work best with TantraFlow?
For speed (Apple Silicon M2, 16GB): llama3.2:3b or gemma2:2b — completes a 4-node workflow in 2–4 minutes. For quality: mistral:7b or llama3.1:8b. For best quality (48GB+ RAM): llama3.1:70b or qwen2.5:72b. We recommend starting with 3b and upgrading once you're comfortable with the platform.
My workflow keeps timing out on Node 1. What's wrong?
This is a model cold-start issue. Ollama takes 30–90 seconds to load a model from disk into memory. TantraFlow sends a warmup ping before Node 1 runs, but if your machine is slow, the model may not finish loading in time. Solutions: (1) Use a smaller model like llama3.2:3b. (2) Keep Ollama running between workflow executions — the model stays in memory for 10 minutes. (3) The timeout is 600 seconds — if you're still seeing it, check your model size vs available RAM.
How do I get PDF output to work?
PDF generation uses WeasyPrint. Install it: pip install weasyprint. On macOS you may also need: brew install pango. If WeasyPrint isn't installed, TantraFlow falls back to saving an HTML file with a note inside — it's still fully printable from your browser using print → save as PDF.
Can I use OpenAI or Anthropic APIs instead of local models?
Yes. In Models → Add Source, choose type OpenAI Compatible and set the base URL to the provider's API endpoint. For OpenAI: https://api.openai.com/v1. Note: you'll need to handle API key injection in your Ollama proxy or modify the executor to pass Authorization headers. This is planned as a first-class feature in a future release.
How do I run TantraFlow in production for a team?
For team deployments, migrate from SQLite to PostgreSQL using the migration package included in this release. The migration guide covers: starting a PostgreSQL container, migrating existing data, resetting sequences, and updating the connection layer. For high concurrency, add pgBouncer as a connection pooler. Run TantraFlow behind nginx for TLS termination and reverse proxy.
The agents produce poor quality output. How do I improve it?
Quality depends primarily on: (1) skills.md clarity — the more specific the role definition and behavioral guidelines, the better. Add explicit output format instructions. (2) Model choice — 3B models produce adequate output for most tasks, but 7B+ models produce noticeably better prose and code. (3) Task instruction per node — set the task_instruction in the Builder Properties panel to give each node a specific, concrete deliverable to produce.
